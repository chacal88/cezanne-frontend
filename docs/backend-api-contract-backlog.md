# Backend/API Contract Backlog

## Purpose

This backlog audits deferred backend, API, provider, and schema blockers across V0-V5 for the next implementation phase.

Rules for this document:
- confirmed facts come from `docs/`, `src/`, and captured visual evidence only;
- unknown payloads and schemas stay unknown until backend/API contracts confirm them;
- Figma drafting status is separate from backend readiness;
- no route family is replacement-approved by this backlog.

## Evidence Baseline

Confirmed runtime/code evidence:
- Auth login uses auth service `/login`, REST `/authenticate`, and GraphQL session enrichment in `src/domains/auth/api/auth-api-adapter.ts`.
- Public auth token helpers call `/user/forgot-password`, `/user/valid-token`, `/user/reset-password`, `/user/first-access`, `/invitation`, `/hiring_company`, and `/recruitment_agency`.
- Cezanne/SAML callbacks exchange provider codes through auth service callback routes before the same bootstrap path.
- Dashboard has an API adapter in `src/domains/dashboard/api/dashboard-api.ts` for GraphQL dashboard aggregates, REST `/notification/notifications`, and GraphQL calendar reads.
- Candidate database has a REST adapter for `/v2/cv` in `src/domains/candidates/database/candidate-database-api.ts`.
- Candidate detail has a GraphQL adapter in `src/domains/candidates/detail-hub/candidate-detail-api.ts`.
- Notifications, inbox, account settings, Jobs, many V3 public/token flows, V4 operations, and V5 SysAdmin/platform routes still use fixture/state/adaptor seams where production contracts are not confirmed.

## Blocking Levels

| Level | Meaning |
|---|---|
| `P0 backend contract blocker` | Production implementation cannot replace fixture/state behavior without a backend/API decision. |
| `P1 schema/payload blocker` | Route-state shell can continue, but field-specific UI, validation, mutation, or parity work must wait. |
| `P2 provider/transport blocker` | Screen-flow can proceed with annotated placeholders, but live integration/provider behavior is blocked. |
| `P3 evidence/parity blocker` | Backend contract may be enough for data work, but replacement approval still needs visual/parity evidence. |

## Recommended Backend Decision Order

1. Close shared auth/token/session contracts first, because every authenticated and public/token phase depends on predictable failure and continuation behavior.
2. Close notification and inbox contracts next, because V0 resolver behavior is a cross-release entry path and later releases add destination kinds.
3. Close Jobs and Candidate aggregates/mutations together, because V1 task overlays and V2 candidate actions share schedule, reject, offer, document, and parent-refresh semantics.
4. Close provider readiness/setup and upload/download provider contracts before field-specific Figma polish for scheduling, publishing, public application, integration forms, contracts, and billing.
5. Close V4/V5 admin schemas after shared provider/payment/report primitives are settled, so platform/admin tables do not fossilize fixture fields.

## Auth-Specific Source Findings

Local source review for `auth-replacement-readiness-package` found these concrete auth behaviors:

- Login error strings are currently emitted by `ms-auth` as message text and interpreted by the legacy frontend through substring checks. This confirms parity targets for activation, approval, Cezanne mandatory, setup required, invalid credentials, and 2FA states, but keeps structured error-code migration as a P0 blocker.
- `v0-auth-token-session-contract-closeout` additionally confirms the source-backed 2FA delivery contract: `ms-auth` sends a six-digit email code, marks verified codes as used, and treats codes as valid for 10 minutes. No source-backed resend endpoint or lockout policy was located.
- Forgot-password decision approved on 2026-04-23: use `mail_not_found`, `mail_sent`, and `mail_error`; preserve the legacy account-not-found disclosure until Product/Security explicitly changes it.
- Reset password decision approved on 2026-04-23: use `token_not_found`, `token_invalid`, `token_accepted`, `token_expired`, and `token_used`; terminal token outcomes return to `/` with safe flash/copy, while password mismatch remains route-local.
- Confirm registration decision approved on 2026-04-23: invalid/expired returns home with safe error copy; `token_valid` with a session token bootstraps the session and hands off to `/dashboard`; `approval_pending` or `token_valid` without a session token returns home with approval-pending copy and no session; `bootstrap_failed` or failed bootstrap after a returned token clears partial continuation and returns home with safe login-retry copy.
- Register/invite decision approved on 2026-04-23: `/register/:token` remains the legacy-backed invite continuation route; `token_valid` may prefill invitation fields, `token_invalid`/`token_expired` return home with invalid invite copy, submit success returns home with success modal/flash, and `/users/invite-token` stays current-app-only until confirmed as a canonical alias. Source/evidence review found legacy `users.invite-token` as an authenticated admin token-link modal, so current `/users/invite-token` has no confirmed public legacy parity target yet.
- Cezanne and SAML callback transport is confirmed in `ms-auth`, including SAML short-lived code exchange and 422 token-not-found behavior.
- Provider callback decision approved on 2026-04-23: preserve legacy launch/callback direction, exclude raw codes from UI/evidence, return provider/missing-code/exchange failures to public safe copy, bootstrap successful tokens to the resolved landing, and clear partial sessions on bootstrap failure.
- Session-loss decision approved on 2026-04-23: no dedicated legacy route/copy was located, so `/session-lost` remains current-app-only with safe expiry copy and explicit separation from `/logout`.
